## 시스템 구성

```
사용자 ──→ Vercel (프론트엔드) ──→ Supabase (DB)
              │                        ↑
              └──→ Railway (백엔드) ────┘
```

| 구성 요소 | 기술 | 역할 |
|-----------|------|------|
| 프론트엔드 | React + TypeScript + Tailwind | 대시보드 화면 |
| 백엔드 | FastAPI (Python) | 업로드, 파이프라인 실행 |
| 데이터베이스 | PostgreSQL (Supabase) | 데이터 저장 + 권한 |
| 배포 | Vercel + Railway | 자동 배포 (git push 시) |

## 대시보드에서 볼 수 있는 것

### 공급망(SCM) 대시보드 — 8개 탭

| 탭 | 보여주는 것 |
|----|-----------|
| 재고 현황 | 창고별/품목별 현재 재고, 유통기한, 판매가능 수량 |
| 재고회전율 | 재고가 얼마나 빨리 소진되는지 (회전비율, 재고일수) |
| 품절 위험 | 현재 판매속도 기준 며칠 뒤 품절되는지 |
| 미입고 발주 | 발주했지만 아직 안 들어온 수량, 예상 도착일 |
| 리드타임 | 발주 → 입고까지 실제 걸리는 기간 분석 |
| 출고/반품 | 일별 출고량, 반품량 추이 |
| 병목 현황 | 공급망에서 문제되는 지점 자동 탐지 |
| 예측 | 수요 예측, 보충 발주 권장량 |

### 손익(P&L) 대시보드 — 5개 탭

| 탭 | 보여주는 것 |
|----|-----------|
| 매출 | 채널별/품목별 매출 |
| 매출원가(COGS) | 판매한 물건의 원가 |
| 공헌이익 | 매출 - 원가 - 변동비 |
| 영업이익 | 공헌이익 - 고정비 |
| 수익성 순위 | 품목/채널별 수익성 비교 |

## 데이터 흐름

```
CSV/엑셀 업로드
    ↓
[raw] 원본 그대로 저장
    ↓
[core] 키 통일, 타입 정리, 품질 검증
    ↓
[mart] 화면용 집계/분석 테이블
    ↓
대시보드에 표시
```

업로드 가능한 데이터 7종:
1. **재고** — 창고별 현재 재고
2. **발주** — 공급사에 발주한 내역
3. **입고** — 실제 입고된 내역
4. **출고** — 출고 내역
5. **반품** — 반품 내역
6. **매출/정산** — 채널별 매출
7. **비용** — 운송비, 창고비 등

## 파이프라인 실행 순서

`run.py --once` 실행 시:

| 단계 | 이름 | 하는 일 |
|------|------|---------|
| 1 | Ingestion | inbox/ 폴더의 파일을 raw 테이블에 적재 |
| 2 | SCM Marts | 재고, 발주, 출고, 반품 등 12개 분석 테이블 생성 |
| 3 | Cost Allocation | 비용을 품목/채널별로 배분 |
| 4 | P&L Marts | 매출, 원가, 이익 등 7개 분석 테이블 생성 |
| 5 | Reconciliation | 대사검증 (OMS↔WMS, ERP↔WMS 등) |
| 6 | Constraints | 병목 지점 자동 탐지 |
| 7 | Coverage | 데이터 커버리지 측정 |
| 8 | Anomaly Detection | 이상치 자동 탐지 (Phase 5 신규) |

## 주요 설정 파일

| 파일 | 용도 |
|------|------|
| `config/thresholds.yaml` | 품절 위험 기준일수, 과재고 기준 등 |
| `config/schema.yaml` | 데이터 스키마 정의 |
| `config/column_aliases.yaml` | 한글 헤더 → 영문 컬럼 매핑 |
| `config/policies/charge_policy.yaml` | 비용 유형별 배분 기준 |
| `config/policies/allocation.yaml` | 비용 배분 규칙 |
| `frontend/.env` | Supabase URL, API 키 |
